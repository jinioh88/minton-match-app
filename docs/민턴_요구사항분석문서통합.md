Phase 1: 서비스 정의 및 타겟 분석 (Service Definition)
가장 먼저 이 서비스가 해결하려는 '페인 포인트(Pain Point)'를 정의합니다.

핵심 가치 정의: "왜 이 서비스가 필요한가?" (예: 급하게 대타가 필요할 때, 내 실력에 맞는 상대를 못 찾을 때)

페르소나 설정: 주 사용자는 누구인가? (예: 2030 직장인 동호인, 클럽 총무, 레슨을 원하는 초보자 등)

플랫폼 범위: 웹과 앱에서 각각 어떤 기능을 중점적으로 제공할 것인가?

Phase 2: 기능 요구사항 나열 (Functional Requirements)
서비스에 들어갈 모든 기능을 브레인스토밍하고 우선순위를 정합니다.

사용자 기능: 회원가입/로그인, 프로필 관리(구력, 라켓 정보), 매칭 글 쓰기, 참여 신청, 채팅, **마이페이지(개설·참여 매칭 수·매칭 내역, 후기 관리, 공지·고객센터 진입, 계정 관리)**.

매칭 로직: 위치 기반 검색, 실력(A~D조) 필터링, 매너 점수 시스템.

관리자 기능: 신고 관리, 배드민턴장 데이터 관리, 공지사항(등록·노출). **이용자 앱**에서는 공지 **목록·상세 조회**, 고객센터(FAQ·문의 안내 등)를 제공한다(`docs/제품백로그.md` Epic 9).

우선순위 선정 (MVP): 당장 출시를 위해 반드시 필요한 기능과 나중에 추가할 기능을 구분합니다.

Phase 3: 비기능 요구사항 정의 (Non-Functional Requirements)
눈에 보이지 않지만 서비스의 품질을 결정하는 요소들입니다.

성능: 매칭 알림이 얼마나 빨리 와야 하는가? (Push 알림 속도)

보안: 개인정보 보호 및 채팅 데이터 암호화.

확장성: 사용자가 갑자기 늘어날 때 서버가 견딜 수 있는 구조인가?

Phase 4: 프로세스 및 사용자 흐름 설계 (User Flow)
사용자가 앱에 접속해서 매칭을 완료하기까지의 여정을 그립니다.

Flowchart 작성: [앱 접속] → [매칭 리스트 탐색] → [상세 확인] → [참여 신청] → [개설자 승인] → [채팅방 생성].

예외 케이스 처리: 매칭이 취소되었을 때, 거절당했을 때의 알림 흐름 등.

Phase 5: 화면 설계 및 상세 명세서 (Wireframing & Spec)
실제 개발자가 보고 코딩할 수 있을 수준의 가이드를 만듭니다.

와이어프레임: 각 화면의 구성 요소(버튼 위치, 입력창 등)를 배치합니다.

데이터 명세: 각 화면에서 보여줘야 할 데이터 항목을 정의합니다. (예: 매칭 상세 페이지에는 '코트 비용 분담 방식' 항목이 필요함)

# 왜만들려하는가?
## 배드민턴을 치고 싶은데 칠 사람이 없다
- 동호회 또는 특정 클럽을 가입하지 못한 사람들은 배드민턴 참여가 어려운 현실이다
- 급수가 낮거나 하면 눈치를 보면서 게임을 쳐야한다. 또 꼰대문화로 잘 쳐주지 않는다.
- 갑자기 배드민턴을 치고 싶은데, 인원이 부족해 치지 못하는 경우가 있다. 

## 누구나 자유롭게 배드민턴을 매칭해서 치는 문화를 만들고 싶다
- 왜 클럽 혹은 동호회에 가입을 해얗만 하는가
- 왜 맨날 똑같은 사람하고만 쳐야히는가?

# 생각한 기능들 정리
## 로그인
- 일반로그인, 카카오, 네이버, 구글, 애플 로그인 필요
- 이름, 나이, 사는곳(현재 위치) 혹은 관심 지역설정, 성별, 급수, 별명, 자기소개, 프로필 사진이 필요함

## 유저 정보
- 유저 정보를 수정할 수 있고 상세정보를 볼 수 있어야 한다
- 다른 사람의 유저 정보를 볼 수 있어야 한다.
- 관심지역은 최대 2군대만 등록할 수 있어야 한다 (매칭 필터에 필요)
- 유저의 패널티 정보나 모임 후기 정보를 볼 수 있어야 한다.

## 마이페이지·계정 (API·기능 보강)
프로필 수정과 별도로 **마이페이지**에서 다음이 가능해야 한다 (프론트 화면이 먼저 있는 경우 **백엔드 API를 맞춰 제공**).

- **활동 지표**: **개설한 매칭 수**와 **참여(확정 기준 등 정책으로 정의)한 매칭 수**를 각각 표시. 기존 Phase 5의「총 매칭 횟수」만으로는 부족하면 **두 지표를 분리**한다.
- **매칭 내역**: 내가 **만든** 매칭 목록, 내가 **참여한** 매칭 목록을 구분해 조회(페이징·상태 필터 등은 구현 시 명세).
- **후기 관리**: 작성한 후기·받은 후기·미작성 대상(종료 매칭 기준) 등 **한 화면에서의 요약·목록**. 기존 후기 작성·조회·익명 공개 규칙과 동일한 정책을 따른다.
- **공지사항·고객센터**: 이용자용 **공지 목록/상세**; 고객센터는 FAQ·문의 안내·외부 링크 또는 경량 API로 제공 (1:1 티켓은 선택).
- **계정 관리**: 로그아웃은 클라이언트 처리 가능하나, **회원 탈퇴·비활성화**, 연동된 소셜 계정 표시 등 **서버 처리가 필요한 계정 수명주기**는 API로 제공한다.

## 매칭
### 매칭 만들기
- 내가 배드민턴 매칭 조건(나의위치, 모임 급수, 모임장소, 시간, 설명, 사진)에 맞게 글을 올리면 해당 조건에 맞게 글을 작성 
- 글을 작성하면 매칭 조건에 맞는 사람에게 알림이 감. 

### 매칭 수정
- 방장으로서 매칭 생성 후 제목, 장소, 일시 등을 잘못 입력했을 때 수정할 수 있어야 한다.
- **정원(maxPeople) 수정 시 제약**: 현재 확정 참여 인원(currentPeople)보다 적게 수정할 수 없다. 예: 4명 모집에 3명 확정 시, 정원을 2명으로 줄이는 것은 불가.
- 모집 중(RECRUITING) 상태에서만 수정 가능 (매칭 완료·취소 후에는 수정 불가).

### 매칭 목록보기
- 사용자의 관심 지역에 따라 해당 매칭 목록이 보여야함. 관심 지역은 한 곳만 지정가능
- 급수, 모임시간, 장소를 필터해서 볼 수 있음

### 매칭 상세보기
- 매칭에 대한 상세 내용을 볼 수 있음

### 매칭 참여
- 사용자는 매칭 상세에 들어가 글을 읽어보고 나에게 조건이 맞으면 매칭 신청을 할 수 있음
- 참여 버튼을 클릭하면 대기열에 들어가고 (참여할때 간단하게 모임의지 멘트를 작성할 수 있음), 매칭신청은 만든 사람에게 알림 또는 화면에 표시가 나와야함. 
- 방장은 매칭 참여자 정보에 대해 볼 수 있어야 하고 해당 조건을 보고 방장은 수락 또는 거부를 할 수 있음

### 매칭 종료(FINISHED)
- 후기 작성·사후 패널티 처리는 **매칭이 종료(FINISHED) 상태**일 때를 전제로 한다.
- 방장이 **수동으로 종료를 확정**할 수 있어야 한다.
- 방장이 종료 처리를 누락하면 후기 작성이 막일 수 있으므로, **모임 시작 시각(`match_date` + `start_time`) 기준으로 일정 시간(예: 24시간)이 지난 매칭**은 시스템이 **스케줄러로 자동 FINISHED** 전환하는 것을 제품 정책으로 둔다. (CLOSED 등 전제 상태는 구현 시 명확히 할 것.)

### 매칭 패널티
- 매칭된 사람끼리 자동으로 채팅방이 열림

### 채팅방
- 매칭에 참여를 눌렀는데, 참여를 안했다면 패널티가 부여됨
- 지각의 경우도 패널티가 부여됨 (해당 기능은 방장이 최종 승인)

### 제재(Suspension) 정책 — 단계별 제재(Graduated Sanction)

패널티 누적 제재는 **매너 있는 커뮤니티** 유지를 위한 핵심 장치다. 단순히 “3회마다 동일 정지”보다 **누적 횟수(또는 가중 점수)에 따른 계단식 제재**가 운영·체감 모두에 유리하다.

#### 권장 단계(스트라이크 기준 예시)

| 단계 | 누적(스트라이크) | 제재 내용 | 비고 |
|:---:|:---:|:---|:---|
| **주의** | 1~2회 | 서비스 이용 가능, 프로필에 **주의** 표시·경고 알림 | 사실상 경고 구간 |
| **제한** | **3회** | **3일간** 모든 매칭 **참여 신청(Apply) 금지** | 방장(매칭 **개설**) 권한은 유지 가능 |
| **정지** | **5회** | **7일간** 서비스 이용 제한(로그인 외 주요 기능 차단 — 구체 범위는 API·화면 정책으로 정의) | 강한 제재 |
| **퇴출** | **10회** | **영구 정지** 또는 계정 비활성화 | 운영자 검토 후 처리 권장 |

- 임계값(3·5·10)·기간(3일·7일)은 **설정 파일**로 조정 가능하게 두는 것을 권장한다.
- **DB**: 제한·정지의 **실효 시각**을 남기기 위해 `Users`에 **`participation_banned_until`**(참여 신청만 금지, 제한 단계용), **`suspended_until`**(넓은 기능 제한, 정지 단계용) 등을 둔다(`docs/ERD.md`). 퇴출은 `account_status` 등 별도 플래그와 병행한다.
- **검증**: `POST /api/matches/{id}/participants` 등 참여 신청 진입 시 `participation_banned_until != null && isAfter(now)`이면 거절. 정지 단계는 `suspended_until` 및(또는) `account_status`로 로그인 후 기능을 제한한다.

#### 패널티 유형별 가중치(선택)

- 노쇼와 지각을 동일 스트라이크로 셀지 정책으로 결정한다.
- **선택**: 지각 1점·노쇼 2점처럼 **`penalty_points` 합산**으로 단계를 판정하면 더 정교하다. 설정에 유형별 가중치를 둔다.

#### 패널티 완화·복구(선택)

- 장기간 무발생 시 스트라이크/점수 **감소**, 또는 **긍정 후기 누적 시 상쇄** 등 복구 기회를 둘 수 있다. 배치·이벤트 설계는 후속 스프린트에서 확장 가능.

#### 프로필·클라이언트

- 사용자가 **제한/정지 해제 시각**을 알 수 있도록 프로필(또는 내 정보) API에 **`participation_banned_until`**, **`suspended_until`**(또는 남은 시간) 노출을 검토한다.

## 후기
- 모임이 끝나면 해당 모임의 특정 사람을 선택해 후기를 남길 수 있다.
- 후기에는 3단계 좋아요, 상세설명, 점수, 해시태그(열정, 매너, 친절, 고수 등등)를 입력할 수 있다

### 후기 상호 익명성·공개 시점
- 동일 모임 참여자끼리 후기를 주고받을 때, **한쪽만 먼저 작성한 사용자가 상대의 점수·내용을 즉시 확인하면 보복성 후기**로 이어질 수 있다.
- **공개 규칙(권장)**: (1) 두 사용자가 **서로에 대한 후기를 모두 제출했거나**, (2) 매칭 **종료(FINISHED) 시점부터 정한 작성 유예 기간(예: 3일)이 경과**한 뒤에만, 상대가 남긴 후기의 **구체적 내용**(점수, 텍스트, 해시태그 등)을 열람할 수 있다. 그 전에는 **「작성 완료」 여부만** 노출하고 내용은 마스킹한다.
- 후기 목록·상세 **조회 API**는 조회자·작성자·피평가자 관계와 위 시점 규칙에 따라 응답 필드를 필터링한다.

### 프로필 평점(rating_score) 산정 정책
- **신규 가입자**의 `rating_score`는 **0**으로 둔다(받은 후기 없음·미평가). 무후기인데 5점으로 보이지 않게 한다. 클라이언트는 0일 때 "평가 없음" 등으로 표시할 수 있다.
- 후기 **점수만 단순 산술 평균**으로 `rating_score`를 갱신하면, 후기 **개수가 적은 사용자**와 **많은 사용자** 간 신뢰도 비교가 왜곡될 수 있다.
- **가상 후기(프라이어)를 반영한 가중 갱신**을 쓴다. 가상 개수 M·프라이어 평균 priorMean(예: 5.0, **내부 가중용**)을 설정에 두고, 실제 받은 후기 수(이번 반영 직전) n, 직전 저장 평점 R, 새 점수 s에 대해 **`n = 0`(첫 후기)**: `R' = (M × priorMean + s) / (M + 1)`**, **`n ≥ 1`**: `R' = (R × (M + n) + s) / (M + n + 1)`**로 갱신한다. 동일 취지의 **베이지안 평균**도 사용 가능하다. M·priorMean 등 상수는 설정 파일로 관리한다.

## 1. 요구사항 카테고리화 (User Story 중심)

### 👤 사용자 및 프로필 (User Management)

* **소셜 로그인:** 카카오, 네이버, 구글, Apple ID 연동.
* **프로필 설정:** 닉네임, 사진, **주력 라켓**, **선호 플레이 스타일**(드라이브, 공격형 등), 급수(A~D/초심).
* **지역 기반:** 관심 지역(최대 2곳) 설정 → 이 정보가 매칭 피드의 기본값이 됨.
* **신뢰도 지표:** 과거 후기 기반 해시태그(매너, 열정 등) 및 노쇼(No-show) 이력 표시.
* **마이페이지·계정:** 개설·참여 매칭 수 및 매칭 내역, 후기 관리 허브, 공지·고객센터, 탈퇴 등 계정 관리 — 상세는 `## 마이페이지·계정 (API·기능 보강)` 및 Epic 9.

### 🏸 매칭 시스템 (Matching Core)

* **호스트(방장) 기능:** 매칭 생성, **매칭 정보 수정**(제목·장소·일시 등), 신청자 프로필 확인 후 **수락/거절**, 매칭 확정.
* **게스트(참여) 기능:** 매칭 리스트 필터링(거리순, 시간순, 급수순), 참여 신청 시 '한마디' 작성.
* **자동 알림:** 내 관심 지역이나 내 급수에 맞는 새로운 매칭이 올라오면 Push 알림.

### 💬 소통 및 사후 관리 (Communication & Feedback)

* **확정자 전용 채팅:** 방장이 수락한 인원들만 자동으로 입장되는 채팅방.
* **페널티 시스템:** 방장이 '노쇼'나 '지각' 체크 시 기록되며, **단계별 제재(주의·참여 제한·정지·퇴출)** 로 이어질 수 있다(`생각한 기능들 정리 > 제재(Suspension) 정책`).
* **상호 리뷰:** 모임 종료 후 참여자들끼리 매너 점수 및 칭찬 해시태그 부여.
* **후기 공개 시점:** 서로 후기를 모두 작성했거나 유예 기간 경과 후에만 상대 후기 **내용**을 열람(보복 방지). 상세는 `생각한 기능들 정리 > 후기` 절 참고.

## 2. 기획의 완성도를 높일 '디테일' 제안

기존 아이디어에 아래 내용들을 한 뼘만 더 보태면 훨씬 강력한 서비스가 될 것 같습니다.

* **급수 검증 로직:** 스스로 입력한 급수(A~D)는 주관적일 수 있습니다. '검증된 급수'와 '자칭 급수'를 구분하거나, 후기 점수가 쌓이면 시스템이 급수를 제안하는 방식은 어떨까요?
* **코트비 정산 기능:** 배드민턴은 코트비와 셔틀콕 분담이 민감합니다. 매칭 생성 시 '1/N 정산', '방장 부담', '게스트 부담' 등을 명확히 선택하게 하면 분쟁을 줄일 수 있습니다.
* **대기열 시스템 (하이브리드 타임아웃):** 매칭이 취소되었을 때를 대비해 '예비 번호'를 받는 기능. 자리가 생기면 대기 1번에게만 15분 내 수락 기회를 부여하고, 2시간 미만이면 긴급 선착순으로 전환하여 매칭 성공률을 높입니다.

## 3. 궁금해하신 'LoL 방식의 자동 매칭'에 대하여

이 아이디어는 **매우 흥미롭지만, 현실적인 제약**을 먼저 고려해야 합니다.

* **장점:** "지금 바로 게임 한 판"을 원하는 유저에게 최고의 경험을 줍니다. 실력 밸런스를 시스템이 맞춰줄 수 있습니다.
* **단점(장벽):** 게임과 달리 '물리적 장소(코트)'가 확보되어야 합니다. 누군가 코트를 예약해둔 상태에서 사람만 매칭할지, 아니면 장소까지 시스템이 매칭할지가 관건입니다.
* **제언:** 처음에는 현재 구상하신 **'게시판형 매칭'**으로 유저 풀을 확보한 뒤, 특정 지역에 사용자가 밀집되었을 때 **'퀵 매칭(Quick Match)'** 기능을 베타 서비스로 도입해 보시는 것을 추천합니다.

### Phase 4: 사용자 흐름 (User Flow) 상세 설계

사용자의 행동을 **[매칭 탐색 → 참여 신청 및 대기 → 확정 및 소통]**의 3단계로 나누어 정의합니다.

#### 1. 매칭 탐색 및 필터링 (Discovery)

* **관심 지역 자동 로딩:** 앱 실행 시 사용자가 설정한 **최대 2곳의 관심 지역** 내 매칭 리스트가 우선 노출됩니다.
* **상세 필터링:** 사용자는 '오늘/내일', '오전/오후', 'A~D조' 등의 필터를 적용해 리스트를 좁힙니다.
* **방장 신뢰도 확인:** 리스트에서 방장의 **후기 점수(별점 및 '매너', '고수' 등의 해시태그)**를 먼저 확인하여 믿을만한 모임인지 판단합니다.

#### 2. 참여 신청 및 대기열 시스템 (Application & Queue)

* **신청 시 멘트 작성:** 사용자가 '참여하기'를 누를 때, 방장에게 보낼 짧은 인사말(예: "공격형 D조입니다! 열심히 칠게요.")을 작성합니다.
* **실시간 알림 (방장):** 방장에게 "새로운 참여 신청이 도착했습니다"라는 푸시 알림이 갑니다.
* **인앱 알림 (신청자):** 방장이 **수락**하면 신청자에게 확정 안내(`PARTICIPATION_ACCEPTED`), **거절**하면 거절 안내(`PARTICIPATION_REJECTED`)가 인앱 알림으로 저장된다(푸시는 Sprint 6 이후 인프라와 연동 가능).

> **TODO (구현 범위 정합):** 현재 백엔드(Sprint 5)는 알림을 **`notifications` 테이블 저장 + REST 조회**로만 제공하며, 클라이언트는 **폴링·화면 진입 갱신**으로 수신한다(`common-docs/api/Sprint5-API.md` §1.3). 위「실시간」「푸시」는 **목표 UX**이며, **모바일 푸시(FCM/APNs)·서버主도 실시간 전달**은 아직 없다. 후속 작업은 `docs/Sprint6-할일.md` Step 3·`docs/스프린트플래닝.md` TODO로 추적한다.
* **방장의 선택 (수락/거절):** 방장은 신청자의 프로필(후기 점수, 주로 쓰는 라켓 등)을 보고 수락 여부를 결정합니다.
* **대기열(Queue) 작동:** 정원이 가득 찬 경우, 사용자는 **'대기 신청'**을 할 수 있습니다.
* **하이브리드 타임아웃 시스템 (Hybrid Timeout System):**
  * **1단계 순차 기회**: 확정 인원 취소 시 대기 1번에게만 "참석 기회가 생겼습니다! 15분 내에 수락해 주세요." 알림. 해당 자리는 **예약 상태**로 다른 사람이 챌 수 없음.
  * **2단계 타임아웃**: 15분(설정 가능) 내 응답 없거나 거절 시 → 기회가 대기 2번에게 이전. 응답 안 한 1번은 대기 상태 해제.
  * **3단계 긴급 선착순**: 경기 시작 2시간 미만 남은 시점에서 자리가 생기면 → 대기열 전체에게 동시 알림, **먼저 수락한 사람**이 확정 (매칭 깨짐 방지).



#### 3. 매칭 확정 및 채팅 (Match Confirmed)

* **채팅방 자동 개설:** 방장이 '수락'을 완료하는 즉시, 해당 매칭 멤버들만 모인 **전용 채팅방**이 생성됩니다.
* **최종 정보 공유:** 채팅방 상단에 장소(지도), 시간, 코트비 정산 방식이 공지로 고정됩니다.
* **패널티 트리거:** 모임 시간 직전 취소나 무단 불참 발생 시, 방장은 해당 유저를 신고하거나 패널티를 부여할 수 있는 권한을 가집니다.

### 🛠️ 예외 케이스 처리 (Exception Flow)

기획의 완성도를 위해 아래 상황들에 대한 흐름도 함께 고려했습니다.

| 상황 | 시스템 처리 방식 |
| --- | --- |
| **방장이 거절했을 때** | 신청자에게 인앱 알림 `PARTICIPATION_REJECTED`(및 푸시가 켜진 경우 동일 메시지) 후, UI에서 리스트 등으로 이동 유도. |
| **방장이 모임을 취소했을 때** | 매칭이 `CANCELLED`가 되면 **ACCEPTED 확정 참가자**에게만 인앱 알림 `MATCH_CANCELLED`를 발송한다. **취소를 실행한 방장 본인은 알림 수신 대상에서 제외**한다. (PENDING/WAITING/RESERVED 등은 MVP 범위에서 제외 — `docs/Sprint5-할일.md` 정책과 동일.) |
| **확정 인원 취소 (경기 2시간 이상 남음)** | 대기 1번에게만 "참석 기회가 생겼습니다! 15분 내에 수락해 주세요." 푸시 알림. 자리는 **예약 상태**로 보호. |
| **15분 내 미응답/거절** | 기회가 대기 2번에게 자동 이전. 1번은 대기 상태 해제. |
| **확정 인원 취소 (경기 2시간 미만)** | **긴급 모드**: 대기열 전체에게 동시 알림, 선착순으로 먼저 수락한 사람 확정. |
| **참여자가 지각했을 때** | 방장이 채팅방 내에서 '지각 체크' 시 해당 유저의 프로필 점수에 자동 반영. |
| **매칭 수정 시 정원 축소** | maxPeople을 currentPeople보다 낮게 수정 불가. 예: 4명 모집에 3명 확정 시 → 정원 최소 3명(방장 포함 4명) |

### 1. 홈 / 매칭 목록 화면 (Main Feed)

사용자가 앱을 켜자마자 보게 될 가장 중요한 화면입니다.

* **상단바:** 관심 지역 선택(최대 2곳 전환), 알림 아이콘(종소리).
* **필터 섹션:** 날짜(오늘/내일/달력), 실력(A~D), 성별, 시간대.
* **매칭 카드(리스트 아이템):**
* **제목:** 예) "일요일 오전 10시 같이 치실 분!"
* **핵심 정보:** 일시, 장소(구 이름), 모집 인원 현황(현재/총원).
* **태그:** 목표 급수(D조 이하 등), 성별 제한 유무.
* **방장 정보:** 방장 닉네임 및 **후기 별점/매너 점수.**


* **플로팅 버튼(FAB):** 매칭 만들기(+) 버튼.

### 2. 매칭 상세 화면 (Matching Detail)

참여 결정을 내리는 최종 관문입니다.

* **헤더:** 방장 프로필 요약(사진, 닉네임, 후기 점수). **방장 본인일 경우 [수정] 버튼 노출** (모집 중 상태에서만).
* **본문:**
* **매칭 정보 요약 카드:** 지도(장소), 정확한 시간, 예상 비용(코트비/콕비).
* **상세 설명:** 방장이 쓴 본문 내용.
* **참여자 현황:** 현재 확정된 인원의 프로필 목록.
* **대기열 현황:** 현재 대기 중인 인원 수.


* **하단 고정바:** [참여 신청하기] 또는 [대기 신청하기] 버튼. (이미 신청했다면 '수락 대기 중' 상태 표시)

### 3. 매칭 만들기/수정 화면 (Create & Edit Match)

방장이 정보를 입력하는 폼 화면입니다. 수정 모드에서는 기존 값을 기본값으로 표시합니다.

* **입력 항목:**
* 제목 및 모임 상세 설명.
* 장소 선택 (지도 API 검색 연동).
* 날짜 및 시간 선택 (피커 도구).
* 모집 인원 설정 (최소/최대).
* 희망 급수 선택 (A~D, 초심).
* 비용 정산 방식 (1/N, 방장 부담 등).


* **완료 버튼:** [매칭 등록하기] (생성 시) / [수정 완료] (수정 시).
* **수정 시 제약:** 모집 인원(maxPeople)을 현재 확정 인원(currentPeople)보다 낮게 설정할 수 없다.

### 4. 채팅 및 매칭 관리 (Chat & Management)

확정된 인원끼리 소통하는 공간입니다.

* **채팅방 상단:** 해당 매칭 요약 정보(일시, 장소) 고정 탭.
* **채팅창:** 실시간 메시지 대화.
* **방장 전용 메뉴:** [참여 확정], [매칭 취소], **[지각/노쇼 체크]** 버튼.
* **공통 메뉴:** [지도 보기], [신고하기], [나가기].

### 5. 내 프로필 / 마이페이지 (My Page)

나의 신뢰도를 증명하고 관리하는 화면입니다.

* **상단:** 프로필 사진, 닉네임, 수정 버튼.
* **나의 지표:**
  * **활동 요약:** **개설한 매칭 수**, **참여(확정)한 매칭 수**를 각각 표시한다. (단일「총 매칭 횟수」만으로 두면 집계 정의가 모호하므로 UI·API에서 분리한다.)
  * 받은 칭찬 해시태그(예: #매너왕 #열정배드민턴).
* **후기 점수:** 3단계 좋아요 비율 혹은 별점.
* **후기 관리(메뉴/섹션):** 내가 쓴 후기·받은 후기·작성 필요한 후기(종료된 매칭 기준) 요약 및 목록 연결. (도메인 규칙은 `## 후기` 및 익명 공개 정책과 동일.)
* **정보 설정:** 주로 쓰는 라켓, 선호 지역(2곳), 급수 설정.
* **매칭 내역:** 내가 만든 매칭 / 내가 참여한 매칭 **각각 목록** (필터·페이징은 데이터 명세에서 정의).
* **공지사항 / 고객센터:** 서비스 공지 조회, FAQ·문의 안내·약관 링크 등.
* **계정 관리:** 탈퇴(또는 비활성화) 요청, 연동 정보 확인(표시 수준은 정책에 따름).

### 1. 와이어프레임 (레이아웃 배치)

단순히 "버튼이 있다"가 아니라, **"어느 위치에 어떤 크기로"** 있는지를 묘사해야 합니다.

> **예시 (매칭 목록 화면):**
> * **상단:** 좌측에 '관심 지역' 드롭다운, 우측에 '종 아이콘'.
> * **중앙:** 세로 스크롤 형태의 카드 리스트.
> * **하단:** 탭바(홈, 채팅, 프로필) 고정.
> 
> 

### 2. 데이터 상세 명세 (Data Spec)

개발자는 데이터의 **성격과 제약 조건**을 알아야 DB를 설계하고 코딩할 수 있습니다.

| 항목명 | 데이터 타입 | 필수 여부 | 비고 |
| --- | --- | --- | --- |
| **매칭 제목** | String(50자 이내) | 필수 | 목록 노출 시 말줄임표 처리 |
| **모집 인원** | Integer(2~12) | 필수 | 최소 2명 이상 설정 가능 |
| **코트비 분담** | Enum (선택형) | 필수 | 1/N, 방장부담, 게스트부담 중 택 1 |
| **방장 별점** | Float (0.0~5.0) | 자동 | 유저 DB에서 실시간 로드 |

### 3. 상태값 및 정책 (Business Logic)

버튼을 눌렀을 때 일어나는 '경우의 수'를 명시해야 합니다.

* **버튼 정책:** 인원이 꽉 차면 [참여 신청] 버튼이 [대기 신청]으로 자동 변경됨.
* **권한 정책:** 방장에게만 [참여 수락/거절] 버튼이 노출됨.
* **매칭 수정 정책:** 모집 중(RECRUITING) 상태에서만 수정 가능. 정원(maxPeople) 수정 시 `maxPeople >= currentPeople` 검증 필수.
